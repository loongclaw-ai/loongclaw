# Extensions

Use this page when you want to build a Loong extension without guessing which
surface is real on the current branch.

This page is intentionally concrete. It describes the extension seams that are
already wired, how to scaffold them, and how to validate them through the same
bounded routes the runtime uses today.

## Choose An Extension Family

| If you need to... | Use | Why |
| --- | --- | --- |
| ship reusable prompt or workflow guidance | managed skills | installable, inspectable, no runtime bridge required |
| add a governed runtime package without host-level trust | governed plugin package | manifest-first, policy-shaped, bounded by bridge/runtime support |
| declare host hooks or shell-first TUI surfaces | trusted host extension | explicit high-trust lane with bounded probes plus runtime-managed TUI execution |

## Current Trusted Host Boundary

Today, trusted host authoring is deliberately narrow:

- runnable `process_stdio` packages only
- explicit `--source-language`
- explicit host-hook and TUI-surface declarations
- bounded probe execution plus runtime-managed TUI execution, not implicit in-process execution

Current built-in trusted shell-first TUI surfaces:

- `command_palette`
- `settings_flow`
- `startup_onboarding`

Trusted host packages can also declare additional lowercase surface identifiers
such as `sidebar_widget`. Those custom surfaces stay inspectable and runnable
through the same trusted runtime lane even when they do not yet have dedicated
first-party UI affordances.

When a trusted surface needs stronger self-description, add
`loong_extension_tui_surface_specs_json` so Loong can reuse shared labels,
summaries, sample payloads, and operator hints across docs and operator
surfaces.

Trusted host hooks can do the same through
`loong_extension_host_hook_specs_json`.

Governed native runtime packages can do the same for
`extension/event`, `extension/command`, and `extension/resource` through
`loong_extension_method_specs_json`.

## Fast Path: Scaffold A Trusted Host Extension

### Example: host hook

```bash
loong plugins init ./weather-host \
  --plugin-id weather-host \
  --bridge-kind process_stdio \
  --source-language js \
  --host-hook turn_start \
  --capability observe_telemetry
```

### Example: TUI surface

```bash
loong plugins init ./weather-palette \
  --plugin-id weather-palette \
  --bridge-kind process_stdio \
  --source-language js \
  --tui-surface command_palette
```

The scaffold writes:

- `loong.plugin.json`
- `README.md`
- runnable local stub files such as `index.js`, `index.py`, `main.go`, or
  `src/main.rs`

It also prints a bounded smoke command so you can validate the package
immediately.

Scaffolded native extensions keep `invoke_connector` as the baseline capability.
Use repeated `--capability <name>` flags only to declare additive capability
intent.

## Validate The Package

```bash
loong plugins inventory --root ./weather-host
loong plugins doctor --root ./weather-host --profile sdk-release
```

## Probe The Runtime Surface

### Host hook

```bash
loong plugins invoke-host-hook \
  --root ./weather-host \
  --plugin-id weather-host \
  --hook turn_start \
  --payload '{"turn_id":"demo-turn"}' \
  --allow-command node
```

### TUI surface

```bash
loong plugins invoke-tui-surface \
  --root ./weather-palette \
  --plugin-id weather-palette \
  --tui-surface command_palette \
  --payload '{"query":":ext"}' \
  --allow-command node
```

## Live Shell-First Routes

If runtime plugins are enabled inside the shell-first TUI, Loong exposes the
same extension truth directly:

- `/extensions`
- `/extensions <plugin-id>`
- `/extensions run <plugin-id> <surface>`

Current live routes already give richer first-party affordances to:

- `command_palette`
- `settings_flow`
- `startup_onboarding`

Those routes now execute through the runtime-managed trusted host lane. They do
not imply a general in-process TUI executor contract.

Any other valid declared trusted TUI surface identifier can still execute
through `loong plugins run-tui-surface` and `/extensions run <plugin-id> <surface>`.

If `[runtime_plugins].enabled = true` and explicit roots are omitted, the
current branch also auto-discovers the standard local roots first:

- `./.loong/extensions`
- `~/.loong/agent/extensions`

If the same `plugin_id` exists in both places, `.loong/extensions/` wins and
the global package is treated as a shadowed duplicate in inventory and runtime
operator surfaces.

When that conflict exists, Loong's operator surfaces are expected to surface:

- the winning project-local package path
- the shadowed global package path
- review commands on `loong status`
- the same discovery actions in `loong doctor --json`
- a manifest comparison flow via `git diff --no-index`

## What Is Real Today

You can rely on:

- trusted-host manifest declarations
- shared declaration parsing in the kernel
- daemon-side bounded probe commands
- daemon-side runtime-managed `loong plugins run-tui-surface`
- shell-first live routing onto that runtime-managed command
- public and source-facing docs that describe the same current surface

You should not assume:

- arbitrary in-process TUI execution
- provider middleware parity with Pi
- tool middleware parity with Pi
- broad bridge-family parity beyond the current process_stdio path
