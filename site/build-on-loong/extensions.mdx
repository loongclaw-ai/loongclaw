# Native Extensions

Use this guide when you want **Pi-style local extension ergonomics** without giving up Loong's governed runtime path.

Loong's native extension lane is:

- **manifest-first**
- **process-bridge governed**
- **operator-inspectable before execution**

The goal is simple:

1. scaffold a package,
2. write a small local runtime file,
3. validate and inspect it,
4. smoke-test it through the same governed bridge the runtime uses later.

## What Loong means by a native extension

A native extension is a **governed plugin package** whose manifest declares a small host-facing runtime contract such as:

- `extension/event`
- `extension/command`
- `extension/resource`

The current shortest authoring lane is a `process_stdio` package.

## Auto-discovery locations

Loong can auto-discover runtime plugin packages from:

- `.loong/extensions/` — project-local
- `~/.loong/agent/extensions/` — global

Use explicit `runtime_plugins.roots` only when you want to override that
default discovery contract.

If the same `plugin_id` exists in both places, `.loong/extensions/` wins and
the global package is treated as a shadowed duplicate in inventory and runtime
operator surfaces.

When that conflict exists, Loong's operator surfaces are expected to surface:

- the winning project-local package path
- the shadowed global package path
- review commands on `loong status`
- the same discovery actions in `loong doctor --json`
- a manifest comparison flow via `git diff --no-index`

## Scaffold a package

Start with `plugins init`.

### Python

```bash
loong plugins init ./weather-python \
  --plugin-id weather-python \
  --provider-id weather \
  --connector-name weather-stdio \
  --bridge-kind process_stdio \
  --source-language py \
  --summary "Python weather extension"
```

### JavaScript

```bash
loong plugins init ./weather-js \
  --plugin-id weather-js \
  --provider-id weather \
  --connector-name weather-stdio \
  --bridge-kind process_stdio \
  --source-language js \
  --summary "JavaScript weather extension"
```

For `process_stdio`, Loong now scaffolds:

- `loong.plugin.json`
- `README.md`
- a runnable `index.py`, `index.js`, or `index.ts` stub

The generated manifest also includes the native extension metadata keys needed for inventory and audit surfaces, including:

- `loong_extension_contract`
- `loong_extension_facets_json`
- `loong_extension_methods_json`
- `loong_extension_events_json`
- `loong_extension_host_actions_json`
- `loong_extension_tui_surfaces_json`

The package manifest also carries explicit capability declarations. The current
native extension scaffold defaults to `InvokeConnector`, and inventory/runtime
operator surfaces should expose that capability truth before execution.

It also declares the current public extension family and trust lane:

- `governed_native_runtime_extension`
- `governed_sidecar`

The scaffold also reserves:

- `loong_extension_host_hooks_json=[]`
- `loong_extension_tui_surfaces_json=[]`

If you keep that field empty, the package stays on the current governed
sidecar lane.

If you pass one or more `--host-hook` values at scaffold time, Loong now
switches the package onto:

- `trusted_host_extension`
- `trusted_host`

and the smoke path changes to:

- `loong plugins invoke-host-hook`

The current trusted host lane is still bounded:

- read-only hooks only
- `process_stdio` only
- daemon-owned lifecycle seams first

The same trusted-host lane also accepts typed shell-first TUI declarations for:

- `command_palette`
- `settings_flow`
- `startup_onboarding`

### Trusted host scaffold

```bash
loong plugins init ./weather-host-js \
  --plugin-id weather-host-js \
  --provider-id weather \
  --connector-name weather-host-stdio \
  --bridge-kind process_stdio \
  --source-language js \
  --host-hook turn_start \
  --host-hook turn_end
```

TUI-oriented scaffold:

```bash
loong plugins init ./weather-host-ui \
  --plugin-id weather-host-ui \
  --provider-id weather \
  --connector-name weather-host-ui \
  --bridge-kind process_stdio \
  --source-language js \
  --host-hook turn_start \
  --tui-surface command_palette
```

Probe one declared hook:

```bash
loong plugins invoke-host-hook \
  --root "./weather-host-js" \
  --plugin-id weather-host-js \
  --hook turn_start \
  --payload '{"turn_id":"demo-turn"}' \
  --allow-command node
```

These TUI declarations are contract-first today: Loong inventories and
validates them on the trusted host lane, while live shell-surface dispatch
remains a separate follow-up seam.

Bounded TUI probe:

```bash
loong plugins invoke-tui-surface \
  --root "./weather-host-ui" \
  --plugin-id weather-host-ui \
  --tui-surface command_palette \
  --payload '{"query":":ext"}' \
  --allow-command node
```

Automatic trusted-host dispatch currently covers:

- `turn_start`
- `turn_end`
- `session_start`
- `session_shutdown`

## Understand the generated stub

The generated runtime file is intentionally small.

It already handles the smallest useful host-facing surface:

- `extension/event`
- `extension/command`
- `extension/resource`

That makes it good enough for a first smoke test while staying easy to replace with your real implementation.

## Validate and inspect the package

Run the author-facing governance surfaces first:

```bash
loong plugins doctor --root "./weather-python" --profile sdk-release
```

Then inspect the package truth:

```bash
loong plugins inventory --root "./weather-python"
```

These surfaces tell you:

- what Loong recognized,
- whether setup is incomplete,
- which extension declarations are visible before execution,
- and what operator actions or remediation remain.

## Smoke-test the extension runtime

Use the explicit smoke path:

```bash
loong plugins invoke-extension \
  --root "./weather-python" \
  --plugin-id weather-python \
  --method extension/event \
  --payload '{"event":"session_start"}' \
  --allow-command python3
```

Or for JavaScript:

```bash
loong plugins invoke-extension \
  --root "./weather-js" \
  --plugin-id weather-js \
  --method extension/event \
  --payload '{"event":"session_start"}' \
  --allow-command node
```

For TypeScript:

```bash
loong plugins invoke-extension \
  --root "./weather-ts" \
  --plugin-id weather-ts \
  --method extension/event \
  --payload '{"event":"session_start"}' \
  --allow-command node
```

The TypeScript template runs through `node --experimental-strip-types index.ts`
behind the governed process bridge.

For Go:

```bash
loong plugins invoke-extension \
  --root "./weather-go" \
  --plugin-id weather-go \
  --method extension/event \
  --payload '{"event":"session_start"}' \
  --allow-command go
```

For Rust:

```bash
loong plugins invoke-extension \
  --root "./weather-rust" \
  --plugin-id weather-rust \
  --method extension/event \
  --payload '{"event":"session_start"}' \
  --allow-command cargo
```

This is intentionally explicit: the smoke probe only runs when you allow the process command on the CLI.

## Suggested iteration loop

1. `plugins init`
2. edit `loong.plugin.json`
3. replace the generated `index.py` / `index.js`
4. `plugins doctor`
5. `plugins inventory`
6. `plugins invoke-extension`

Repeat until:

- declarations look right,
- the smoke probe returns the payload you expect,
- and doctor stops surfacing author-facing blockers you care about.

## What this does not promise yet

This guide is for the **non-TUI authoring lane**.

It does **not** cover:

- TUI extension discovery
- TUI widget or panel hosting
- TUI render-host integration

Those are intentionally separate from the public authoring path.
